#### HIGH — `mdma_client` is a "god client" component

`mdma_client` aggregates ALL IPC client components into one super-component. Both `mdma-cli` and `mdma-tui` depend on it, meaning they are coupled to ALL services — even ones they may not actually use at runtime. This is the "fat component" anti-pattern.

In polylith, this wiring role belongs to a **project** (or, in a pinch, a base). As a component it forces every consumer to pull in the full IPC surface.

#### HIGH — No `projects/` → no deployment flexibility

There is no way to define "ship only these components for this deployment context." Polylith's key benefit — the ability to compose different deployables from the same component pool — is entirely unrealized. Examples of deployables that would benefit from project definitions:
- `mdma-server` (gateway + library + acid + audio + playback + bandcamp services)
- `mdma-client-only` (cli + tui, no servers)
- `mdma-headless` (services only, no UI)

#### MEDIUM — Bases without `[[bin]]` entries

A polylith base IS an entry point. These four bases have no `[[bin]]` defined:

| Base | Status | Issue |
|------|--------|-------|
| `mdma-acid` | No `[[bin]]` | Library crate in bases/ — binary? |
| `mdma-library` | No `[[bin]]` | Library crate in bases/ |
| `mdma-console` | No `[[bin]]` | Axum server, no binary target |
| `beacon` | No `[[bin]]` | Axum server, no binary target |

`mdma-console` and `beacon` run HTTP servers (Axum) but expose no binary — they must be reachable through some other mechanism (possibly a workspace-level main or a runner script). This is a misclassification or a missing `[[bin]]` entry.

#### MEDIUM — `*_client` components do IO (boundary concern)

`acid-client`, `gateway-client`, `library-ipc-client`, `media-client` all wrap NNG sockets — they open connections to external IPC endpoints. In polylith, connecting to external systems is a boundary/infrastructure concern that naturally lives at the base level.

Using components for this:
1. Creates component-on-component dependency chains (gateway-client depends on 5 other components)
2. Means the "IPC wire format" bleeds into the component layer
3. Forces protocol types to be defined in components that client components then re-import

#### LOW — Orphan components (tool-detected)

`audio-metadata`, `date-expression`, `media-client`, `time-primitives` are not referenced by any base. Either they are:
- Work-in-progress (not yet wired)
- Dead code (should be removed)
- Used indirectly via component-on-component deps (which is itself the violation)

Note: `audio-metadata` IS used by `library-service` component and `media-client` IS used by `mdma-client` component — so the orphan report is accurate at the base level but they're reachable via the component dependency chain.

## 3. The "Service Base" Insight

### What the pattern is

Six bases are IPC services: `mdma-gateway`, `mdma-library`, `mdma-acid`, `mdma-audio`, `mdma-playback`, `mdma-bandcamp`. Each almost certainly contains this boilerplate:

```rust
// Duplicated in each service base's main.rs:
tracing_subscriber::fmt()
    .with_env_filter(EnvFilter::from_default_env())
    .init();

let args = Args::parse();

color_eyre::install()?;

let runtime = tokio::runtime::Builder::new_multi_thread()
    .enable_all()
    .build()?;

runtime.block_on(async {
    // signal handling
    tokio::select! {
        _ = signal::ctrl_c() => { /* shutdown */ }
        result = run(args) => { result }
    }
})
```

All of these also use: `nng`, `tracing-subscriber`, `clap`, `color-eyre`, `tokio`.

**This scaffolding should be a `service_runtime` component.**

### What `service_runtime` would contain

```rust
// components/service_runtime/src/lib.rs

pub struct ServiceConfig {
    pub name: &'static str,
    pub version: &'static str,
}

pub fn init_tracing() { /* tracing-subscriber setup */ }
pub fn init_error_handling() { /* color-eyre */ }

pub async fn run_service<F, Fut>(config: ServiceConfig, f: F) -> eyre::Result<()>
where
    F: FnOnce() -> Fut,
    Fut: Future<Output = eyre::Result<()>>,
{
    init_tracing();
    init_error_handling()?;

    tokio::select! {
        _ = signal::ctrl_c() => { Ok(()) }
        result = f() => { result }
    }
}
```

Each service base then becomes thin:

```rust
// bases/mdma_acid/src/main.rs
use service_runtime::run_service;
use acid_protocol::AcidService;

fn main() -> eyre::Result<()> {
    let args = AcidArgs::parse();
    tokio::runtime::Builder::new_multi_thread()
        .enable_all()
        .build()?
        .block_on(run_service(config!(), || async { AcidService::run(args).await }))
}
```

### Why this is a polylith win

1. Signal handling, tracing init, and tokio runtime are infrastructure — they belong in a component
2. Each service base shrinks to a few lines of wiring
3. Changes to startup behavior propagate everywhere from one place
4. Integration tests for `service_runtime` validate startup behaviour once

## 4. Recommended Target Architecture

### Component layer (pure business logic, no IO, no cross-component deps)

**Keep as-is (pure data/type components):**
- `music-primitives`, `storage_primitives`, `time-primitives`
- `playback-primitives` (but remove dep on `music-primitives` — duplicate the types or use a shared crate)

**Protocol components (pure message types — no networking code):**
- `acid_protocol`, `event-protocol`, `gateway-protocol`, `library-ipc-protocol`
- `media-protocol`, `source-protocol`, `stream-source-protocol`

These are currently correct. Keep them.

**Service logic components (pure business logic):**
- `library-search`, `library-service`, `music-facts`, `playback-engine`
- `audio-metadata`, `audio-transcoder`, `bandcamp-api`
- `inbox-utils`, `nng-transport` (infrastructure but stateless)
- `date-expression` (currently orphan — needs wiring)

**New component to create:**
- `service_runtime` — tokio setup, signal handling, tracing init

**Components to dissolve or reclassify:**
- `mdma-client` → dissolve; each base lists its own IPC clients
- `acid-client`, `gateway-client`, `library-ipc-client`, `media-client` → move IPC socket logic into the bases that use them, OR accept as components but enforce they only depend on protocol types + `nng-transport`

### Base layer (entry points / boundaries)

| Base | Action |
|------|--------|
| `mdma-cli` | Remove `mdma-client` dep; explicitly list needed clients |
| `mdma-tui` | Remove `mdma-client` dep; explicitly list needed clients |
| `mdma-console` | Add `[[bin]]` entry |
| `mdma-acid` | Add `[[bin]]` entry; adopt `service_runtime` |
| `mdma-library` | Add `[[bin]]` entry; adopt `service_runtime` |
| `beacon` | Add `[[bin]]` entry; or reclassify as component |
| `mdma-gateway` | Adopt `service_runtime` |
| `mdma-audio` | Adopt `service_runtime` |
| `mdma-playback` | Adopt `service_runtime` |
| `mdma-bandcamp` | Adopt `service_runtime` |

### Project layer (deployment artifacts — to be created)

```
projects/
  mdma-server/       — all service bases (gateway, library, acid, audio, playback, bandcamp)
  mdma-client/       — cli + tui
  mdma-full/         — everything
  mdma-headless/     — server + beacon, no UI
```

Each project's `Cargo.toml` selects which bases (and transitively, which components) are included.

## 5. Open Questions for Brainstorm

1. **Dissolve `mdma_client`?** Each base explicitly lists the IPC clients it needs? Or is there a legitimate "client bundle" pattern?

2. **`service_runtime` scope**: Just tokio + signal handling? Or also NNG socket lifecycle (bind, listen loop, shutdown drain)?

3. **`*_client` components**: Should they move to project-level (wired via Cargo `[patch]` or workspace features)? Stay as components but only depend on protocol types? Move into bases?

4. **Project definition shape**: One project per deployable binary? Or one project = "full system" with feature flags to enable/disable bases?

5. **Plugin model fit**: `source_protocol` enables auto-discovered sources (plugins?). Does socket-autodiscovery belong in `beacon`? Is beacon a discovery service base or a monitoring base?

6. **Protocol + client merge**: Is `acid_protocol` / `acid-client` the right split? Or should they merge into `acid-ipc` with a clean `AcidClient` struct and `AcidService` trait?

7. **Primitive dependencies**: `playback_primitives` depends on `music-primitives`. Is that a legitimate layering (primitive hierarchy) or should both be merged into one `domain-primitives` component?

8. **`library_service` as a component**: It depends on 7 other components. Is this the right level of aggregation, or should it be a base with `library-search`, `music-facts`, etc. as the components it uses?
